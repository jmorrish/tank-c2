DEPLOY COMMAND

# From Windows dev machine
python c:\python\tank-c2\deploy_phase4.py

The script performs these steps:

  1. Connects to Jetson via SSH (paramiko)
  2. Opens SFTP session
  3. Uploads all files to /home/james/YOLOv8-TensorRT/csrc/jetson/detect/
  4. Makes start_tank.sh executable (chmod +x)
  5. Runs cd build && cmake .. && cmake --build . (300s timeout)
  6. Prints build output and exit code

DEPLOY PROCESS

VPS deployment follows the same paramiko SFTP pattern. Upload changed files, then reload pm2:

# Python snippet — deploy server.js (and any other changed files) to VPS
import paramiko, os

ssh = paramiko.SSHClient()
ssh.set_missing_host_key_policy(paramiko.AutoAddPolicy())
ssh.connect("87.106.188.134", username="root", password="CP2Xy3TDzxHGVpcV")
sftp = ssh.open_sftp()

# Upload files
sftp.put(r"c:\python\tank-c2\server.js", "/var/www/tank-c2/server.js")
# ... upload any other changed files ...

# Reload pm2
stdin, stdout, stderr = ssh.exec_command("pm2 reload /var/www/tank-c2/ecosystem.config.js")
print(stdout.read().decode())

sftp.close()
ssh.close()

5. STARTING THE STACK

start_tank.sh is the unified launcher. One command starts all 6 subsystems, one Ctrl+C stops everything.

LAUNCHER COMMANDS

# Full stack (all subsystems)
bash start_tank.sh

# Skip EKF (use until wheel encoders are physically wired)
bash start_tank.sh --no-ekf

# Skip Nav2 planner
bash start_tank.sh --no-nav2

# Skip both
bash start_tank.sh --no-ekf --no-nav2

STARTUP SEQUENCE

#SubsystemWhat It DoesSkippable
1 ydlidar_ros2_driver YDLIDAR X3 360-degree lidar scan publisher No
2 slam_bridge.py ROS2 ↔ TCP bridge (port 9997). Manages SLAM Toolbox lifecycle, publishes /imu/data + /gps/fix, hosts Nav2 action client, dynamic odom→base_link TF No
3 robot_localization EKF + navsat Fuses wheel odometry + IMU. navsat_transform for GPS↔odom conversion --no-ekf
4 Nav2 planner_server SmacPlanner2D on SLAM costmap for obstacle-aware path planning --no-nav2
5 mjpeg_bridge.py MJPEG fan-out relay on port 8080. Subscribes ZMQ 5558 for stereo disparity No
6 yolov8 (C++ app) YOLOv8 detection, person tracking, stereo depth, all comms (Teensys, VPS, SLAM bridge) No

DIRECT C++ APP ONLY

If the ROS2 stack is already running separately:

./yolov8 /home/james/YOLOv8-TensorRT/yolov8n.engine --auto-continue --headless

# With explicit Teensy IPs (these are the compiled defaults)
./yolov8 /home/james/YOLOv8-TensorRT/yolov8n.engine --auto-continue --headless \
    --ptu-ip 192.168.1.177:23 --sensor-ip 192.168.1.178:23
Do NOT run EKF without wheel encoders — EKF has no velocity/position source without them. Always use --no-ekf until encoders are physically wired.

7. GIT WORKFLOW

Git is used on the Windows dev machine only. The Jetson deploy directory is inside an upstream repo and cannot use git.

BRANCH STRUCTURE

BranchPurposeStatus
main Stable releases Base branch for PRs

COMMIT CONVENTIONS

Recent commit messages follow a pattern of short imperative descriptions:

Add system monitor, unified video panel, depth quality control, IP fixes
Fix code review bugs: signal handler, heading EMA, NMEA checksum, fd race
Add behavior architecture, Nav2 path planning, EKF config, unified launcher
Smart camera fallback + stereo lag fix
Fix detection FPS: camera resilience + 28fps recovery

Pattern: [Add|Fix|Split|Drop|Rate-limit] <concise description>

TYPICAL WORKFLOW

# 1. Edit source files locally in c:\python\tank-c2\jetson\

# 2. Deploy to Jetson and build
python deploy_phase4.py

# 3. SSH to Jetson, test
bash start_tank.sh --no-ekf

# 4. Iterate until working, then commit
git add jetson/comms.cpp jetson/include/comms.h
git commit -m "Add heartbeat to control Teensy link"

# 5. Push feature branch
git push origin feature/lidar

# 6. PR into main when stable
gh pr create --base main --head feature/lidar --title "Add lidar + SLAM stack"
The deploy script (deploy_phase4.py) deploys from the local jetson/ directory, not from git.
Always commit after testing on the Jetson, not before.
