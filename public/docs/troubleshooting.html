1. TEENSY CONNECTION ISSUES

Control Teensy: 192.168.1.177:23 — Sensor Teensy: 192.168.1.178:23 — TeensyLink reconnects with 2s backoff.

Symptom Cause Fix
Control/Sensor: connect failed errno=113 (No route to host) Teensy is on wrong subnet. FLIR Tau default IP was 192.168.0.71 — if the Teensy was flashed with a 192.168.0.x address instead of 192.168.1.x, it will be unreachable from the Jetson (192.168.1.167). Verify Teensy sketch has the correct static IP:
IPAddress ip(192, 168, 1, 177); for Control
IPAddress ip(192, 168, 1, 178); for Sensor
Reflash the sketch if wrong. Check with: ping 192.168.1.177
Teensy not reachable — ping times out NativeEthernet was not initialised before UART Serial. Teensy 4.1 has a DMA channel conflict — Ethernet.begin() must run BEFORE any Serial.begin() calls or the Ethernet hardware silently fails to initialise. In the Teensy sketch setup(), ensure:
Ethernet.begin(mac, ip);   // FIRST
// ... then ...
Serial3.begin(115200);     // AFTER Ethernet
This applies to both Control and Sensor Teensy sketches.
Log: Sensor: TCP closed by peer repeating every 2s Teensy sketch accepts a connection but immediately closes it, or the Teensy is rebooting in a loop (power brown-out, watchdog reset). Check Teensy power supply — USB must provide clean 5V. Check sketch for crash in loop(). Connect via Arduino Serial Monitor to see boot messages. TeensyLink will auto-reconnect every 2s.
SENSOR_READY never received — no IMU/GPS data The Sensor Teensy sends SENSOR_READY on new client connection, but the C++ app's connectSensor() immediately sends IMU_ON and IMU_RATE 50 via the onConnect callback. If the Teensy hangs during BNO085 init, it never gets to the network accept loop. Verify BNO085 wiring (PS0=GND, PS1=3V3 for SH2-UART mode). Check that bno.begin_UART() returns true. The IMU must be physically present and powered — begin_UART() blocks for several seconds on failure.
Teensy connects but data is garbled or partial TCP_NODELAY is set on the socket (TeensyLink line 50), which is correct. But if the Teensy sends data without newline terminators, the line parser accumulates indefinitely. Ensure every Teensy output line ends with \n. The C++ rxLoop splits on \n and strips trailing \r. Verify with nc 192.168.1.178 23 from the Jetson to see raw output.
Reconnection loop: connect succeeds but immediately disconnects Teensy NativeEthernet server only accepts one client at a time. If a stale socket from a previous C++ app run is still in TIME_WAIT on the Teensy side, it may reject the new connection. Power-cycle the Teensy to clear stale sockets. The C++ TeensyLink uses a 2s connect timeout with non-blocking connect + select(), so it will retry. Wait ~10s for the previous connection to fully time out.
Quick check: From Jetson terminal: nc -w2 192.168.1.178 23 — you should see SENSOR_READY followed by YPR lines if IMU is enabled. For Control: nc -w2 192.168.1.177 23.

2. CAMERA / VIDEO ISSUES

Stereo camera: 2560x720 MJPEG @ 30fps via V4L2. Detection uses left half (1280x720). FLIR Tau: RTSP at 192.168.1.71.

Symptom Cause Fix
Camera runs at 3fps instead of 30fps Critical V4L2 gotcha: The stereo camera at 2560x720 only supports 30fps in MJPEG mode. If FOURCC is not set to MJPG before setting resolution, or if any FPS other than 30 is requested, V4L2 falls back to raw YUV which maxes out at ~3fps due to USB bandwidth. The open sequence must be exactly: 
cap.set(CAP_PROP_FOURCC, VideoWriter::fourcc('M','J','P','G'));  // FIRST
cap.set(CAP_PROP_FRAME_WIDTH,  2560);
cap.set(CAP_PROP_FRAME_HEIGHT,  720);
cap.set(CAP_PROP_FPS, 30);  // must be exactly 30
The FOURCC must be set before the resolution. Any other FPS value (25, 60, etc.) causes fallback to YUV.
File: stereo_depth.cpp line 125-128, object_detection.cpp line 374-377.
Detection FPS shows ~15fps on first start, recovers to ~28fps after a few seconds V4L2 camera warm-up. The USB camera driver takes 1-3 seconds to stabilise its internal MJPEG encoder after opening. During this period frames are decoded slowly or arrive at irregular intervals. This is normal behaviour. The detection loop in object_detection.cpp starts counting FPS from the first frame. Wait 5-10 seconds for the reported FPS to stabilise. If it stays at 15fps, check that no other process has the camera open (fuser /dev/video*).
Log: cam1: no detection camera found — retrying The detection camera scan (devices 0-7) skips any device that accepts 2560x720 (identifies it as the stereo camera). If the only camera is the stereo cam, or if the detection cam is on /dev/video8+, it will not be found. Use a udev symlink path for the detection camera:
--cam1 /dev/video-detect
Or if running in stereo-fallback mode (no separate detection camera), pass --cam1 "" to use the left half of the stereo camera via StereoDepth frame sharing.
Log: StereoDepth: camera lost — reconnecting 30 consecutive empty grab() calls. The USB camera was physically disconnected, or another process opened the same /dev/video device. Check USB cable. Check no other process has the camera: fuser /dev/video*. The stereo depth thread will auto-reconnect. If both StereoDepth and ObjectDetection try to open the same physical camera, one will fail — ensure they use different devices or use stereo-fallback mode.
Log: RTSP: open failed — retrying in 5s (FLIR thermal) FLIR Tau is on 192.168.1.71 (previously 192.168.0.71). RTSP URL: rtsp://192.168.1.71/z3-2.sdp. If the FLIR is still on the old 192.168.0.x subnet it won't be reachable. Check config.json "cam2_rtsp" matches the FLIR's actual IP:
"cam2_rtsp": "rtsp://192.168.1.71/z3-2.sdp"
Test with: ffprobe rtsp://192.168.1.71/z3-2.sdp
The RTSP thread retries every 5s and reconnects after 10 consecutive read failures. This is non-blocking — the main detection loop runs without the thermal camera.
Log: StereoDepth: cannot open calibration file Calibration XML not found at the path specified in config.json. Default path: /home/james/stereo_calib/stereo_params_cuda.xml
Verify the file exists and contains map1x, map1y, map2x, map2y, P1, and T matrices. Set in config.json:
"stereo_calib_xml": "/home/james/stereo_calib/stereo_params_cuda.xml"
Camera resolution rule: 2560x720 MJPEG only supports 30fps. Requesting any other FPS causes V4L2 to silently fall back to raw YUV at 3fps. Always use CAP_PROP_FPS, 30.

3. SLAM / NAVIGATION ISSUES

YDLIDAR X3 via ROS2 → SLAM Toolbox → slam_bridge.py (TCP :9997) → C++ app. Nav2 SmacPlanner2D for path planning.

Symptom Cause Fix
Lidar not publishing /scan topic Wrong params file. The YDLIDAR X3 requires X3.yaml, NOT ydlidar.yaml. The default ydlidar.yaml has wrong baud rate and channel settings for the X3 model. Use the correct params file: 
ros2 run ydlidar_ros2_driver ydlidar_ros2_driver_node \
  --ros-args --params-file ~/ros2_ws/src/ydlidar_ros2_driver/params/X3.yaml
Required X3 params: baud_rate: 115200, single_channel: true, fixed_resolution: true.
start_tank.sh already uses the correct path (start_tank.sh line 17).
Lidar driver starts but /scan is empty or has NaN ranges YDLIDAR X3 USB device permissions or wrong /dev/ttyUSB* device. Another process may have the serial port open. Check device exists: ls /dev/ttyUSB*
Set permissions: sudo chmod 666 /dev/ttyUSB0
Or use a udev rule. Kill stale processes: pkill -f ydlidar
Verify with: ros2 topic echo /scan --once
SLAM Toolbox not converging — /pose not publishing TF tree is incomplete. SLAM Toolbox requires the chain: odom → base_link → laser_frame. If slam_bridge.py has not yet published the dynamic odom → base_link TF, the scan message filter times out. slam_bridge.py publishes an initial identity odom → base_link TF on startup, then waits 3s before launching SLAM Toolbox (via a timer callback). If SLAM still fails:
ros2 run tf2_ros tf2_echo odom base_link
ros2 run tf2_ros tf2_echo base_link laser_frame
Both must resolve. If not, restart slam_bridge.py.
Nav2 planner_server crashes: "costmap node already exists" Do NOT launch a standalone nav2_costmap_2d node. The planner_server creates its own costmap internally. Launching a separate costmap node causes a name collision. The lifecycle manager in nav2_planner_launch.py only manages ['planner_server']. Remove any standalone costmap node from your launch configuration. The planner's internal costmap reads /scan and /map topics directly.
Nav2 planning fails — planned_path ok:false Nav2 planner not active (lifecycle not configured), or no costmap data (no /scan, no /map), or goal is in an obstacle/unknown cell. Check planner lifecycle state: ros2 lifecycle get /planner_server (should be "active").
Check costmap is receiving data: ros2 topic echo /global_costmap/costmap --once
The C++ Nav2Planner falls back to direct GPS navigation if Nav2 is unavailable — missions will still work but without obstacle avoidance.
Obstacle avoidance not working despite lidar running slam_bridge.py not connected to C++ app. The scan data flows: ydlidar → /scan → slam_bridge.py → TCP :9997 → C++ app. If the TCP link is down, the C++ app gets no lidar data. Check slam_link connection in the C++ log: SLAM bridge connected on 127.0.0.1:9997. If missing, ensure slam_bridge.py is running and listening on port 9997. The SlamLink auto-reconnects every 2s.
TF tree (required): map → odom → base_link → laser_frame
SLAM Toolbox publishes map → odom. slam_bridge.py (or EKF) publishes odom → base_link. Static TF for base_link → laser_frame.

4. EKF / ODOMETRY ISSUES

robot_localization EKF fuses /odom + /imu/data. navsat_transform converts GPS↔odom. Requires wheel encoders.

Symptom Cause Fix
EKF node crashes or produces wild pose estimates on startup Wheel encoders are not physically wired yet. The EKF is configured to fuse /odom (wheel odometry), but without encoders the /odom topic either has zero velocity or is not published. The EKF has no position/velocity source and diverges. Use the --no-ekf flag until wheel encoders are wired: 
bash start_tank.sh --no-ekf
When EKF is skipped, slam_bridge.py publishes odom → base_link dynamically instead. The system works without EKF — it just lacks fused odometry.
navsat_transform_node logs: "Datum recalculation" warnings repeatedly GPS cold start drift. After power-on, the GPS module (BN880) takes 2-5 minutes to get a stable fix. During this time, position and altitude values drift significantly, causing navsat_transform to repeatedly recalculate its datum. This is normal behaviour for the first 2-5 minutes after GPS cold start. Wait for the GPS to stabilise (check sats count — need 6+ for good fix, quality should be 1 or 2). The warnings will stop once altitude/position converge.
Odometry not activating — slam_bridge stays in ODOM_INACTIVE slam_bridge.py requires 5 consecutive fresh encoder reads (age_ms < 200) to transition from INACTIVE to ACTIVE. The SlamLink TX thread sends encoder data every 100ms, but only if age_ms < 200. If encoders are stale, no odom_data messages are sent. Verify encoder data is arriving from the Sensor Teensy:
Check C++ log for ENC <left> <right> lines.
Once 5 fresh reads arrive, slam_bridge transitions to ACTIVE, kills SLAM Toolbox, and restarts it with the odom-enabled params YAML.
GPS↔map coordinate conversion is wrong — waypoints appear offset GPS heading auto-calibration uses an EMA filter comparing GPS course to IMU yaw. If the robot hasn't moved enough (speed < 0.5 knots), the heading offset won't be calibrated, causing GPS→map conversion to be rotated. Drive the robot in a straight line for 5-10 seconds to allow the heading calibration EMA to converge. The mission_moving_knots threshold is 0.5 knots by default (configurable in config.json). Check the slam_bridge.py log for "heading offset" calibration messages.
Do not run EKF without wheel encoders. The EKF has no velocity/position source without them and will produce garbage output. Always use --no-ekf until encoders are physically wired and verified.

7. IMU ISSUES (BNO085)

BNO085 in SH2-UART mode (PS0=GND, PS1=3V3). Body IMU on Sensor Teensy (Serial3). PTU IMU on Control Teensy (Serial5).

Symptom Cause Fix
BNO085 init fails — begin_UART() returns false PS0 and PS1 mode-select pins are not wired correctly. The BNO085 supports multiple protocols: PS0=GND, PS1=3V3 selects SH2-UART mode. Any other combination selects I2C or SPI, causing UART communication to fail. Verify wiring:
PS0 → GND (hardwired, not floating)
PS1 → 3V3 (hardwired, not floating)
Both pins must be connected before power is applied. The mode is latched at boot. Power-cycle the IMU after fixing wiring.
Second BNO085 instance fails — one Teensy reads IMU, the other doesn't Adafruit_BNO08x supports only ONE UART instance per Teensy. The library uses global HAL state for UART communication. You cannot have two BNO085 sensors on two different Serial ports on the same Teensy. Use one BNO085 per Teensy only. The Body IMU is on Sensor Teensy (Serial3), the PTU IMU is on Control Teensy (Serial5). This is the correct architecture — do not try to add a second IMU to either Teensy.
IMU data stops after a few minutes — YPR lines stop appearing The BNO085 performed an internal reset (watchdog, power glitch, etc.) but the sketch did not detect it. After reset, all enabled reports are cleared and must be re-enabled. The Teensy sketch must check bno.wasReset() every loop iteration and re-enable reports: 
if (bno.wasReset()) {
    bno.enableReport(SH2_ROTATION_VECTOR, 20000);  // 20ms = 50Hz
}
Also check for power supply stability — brown-outs cause resets.
IMU init hangs the Teensy — Ethernet never starts Ethernet must init BEFORE UART on Teensy 4.1. NativeEthernet and Hardware Serial both use DMA channels. If Serial.begin() runs before Ethernet.begin(), the DMA allocation conflicts and Ethernet silently fails, or the sketch hangs. Correct init order in setup(): 
Ethernet.begin(mac, ip);       // FIRST - DMA channels allocated
server.begin();
Serial.begin(115200);          // OK after Ethernet
Serial.setTimeout(0);
bno.begin_UART(&Serial3);     // OK after Ethernet
bno.enableReport(SH2_ROTATION_VECTOR, 20000);
Compilation error: "PI was not declared" or "PI redefined" The Adafruit BNO08x library internally uses PI which conflicts with math.h's definition on some platforms. Add before the BNO08x include: 
#undef PI
#define PI 3.14159265358979323846
#include <Adafruit_BNO08x.h>
BNO085 init sequence: Serial.begin(115200) → Serial.setTimeout(0) → begin_UART(&SerialN) → enableReport(SH2_ROTATION_VECTOR, 20000). Check wasReset() every loop and re-enable reports.

8. SLAM_BRIDGE.PY ISSUES

slam_bridge.py: ROS2 node + TCP server (:9997). Manages SLAM Toolbox lifecycle, publishes /imu/data, /gps/fix, odom→base_link TF.

Symptom Cause Fix
slam_bridge.py crashes on startup: AttributeError: '_dyn_tf_broadcaster' Init order bug. _dyn_tf_broadcaster (TransformBroadcaster) must be created before any call to _publish_odom_tf(). If code is added to __init__ that calls _publish_odom_tf() before the broadcaster is assigned, it crashes. In slam_bridge.py, ensure the order in __init__ is: 
# 1. Static TF broadcaster (base_link → laser_frame)
self._tf_broadcaster = StaticTransformBroadcaster(self)
self._tf_broadcaster.sendTransform([...])

# 2. Dynamic TF broadcaster (MUST be before _publish_odom_tf)
self._dyn_tf_broadcaster = TransformBroadcaster(self)

# 3. NOW safe to publish initial identity odom → base_link
self._publish_odom_tf(0.0, 0.0, 0.0, stamp)
Currently correct in the codebase (lines 106, 126, 130). Do not reorder.
Odometry state stuck in INACTIVE despite encoders running The odom state machine requires 5 consecutive fresh reads (age_ms < 200) to transition INACTIVE → ACTIVE. If even one stale read interrupts the sequence, the fresh counter resets. The SlamLink TX thread on the C++ side only sends odom_data when age_ms < 200. Check that encoder data is consistently fresh:
1. On C++ side: verify ENC <left> <right> arrives at 10Hz from Sensor Teensy.
2. slam_bridge.py logs transitions: look for "odom INACTIVE -> ACTIVE".
3. If encoder age is borderline (~200ms), the data may alternate fresh/stale. The check interval is every 100ms from the SlamLink TX thread.
SLAM Toolbox restarts repeatedly (kill/restart loop) Odometry state is flapping between ACTIVE and INACTIVE. Each transition triggers: save map → kill SLAM Toolbox → restart with matching params yaml (odom or no-odom). If encoder data is intermittent, this causes rapid restart cycles. Fix the root cause: ensure encoder data is consistently present or consistently absent. Intermittent encoder connections (loose wire, Teensy brown-out) cause state flapping. The thresholds are: 5 fresh reads to activate (FRESH_RUNS_TO_ACTIVATE), 3 stale reads to deactivate (STALE_RUNS_TO_DEACTIVATE).
GPS data not publishing on /gps/fix topic GPS publishing is throttled to 1Hz in slam_bridge.py (_last_gps_pub_time). Also, the C++ app only forwards GPS data after a valid NMEA checksum passes. If the GPS module is indoors or has no fix (quality=0), no data is forwarded. Take the robot outdoors for GPS fix. Check Sensor Teensy is forwarding NMEA: nc 192.168.1.178 23 should show $GNGGA lines. The comms.cpp NMEA parser validates checksums and requires quality > 0 before setting GPS data.
slam_bridge.py: "nav2_msgs not installed — path planning disabled" The nav2_msgs package is not installed in the ROS2 environment. Install Nav2 dependencies: 
sudo apt install ros-humble-nav2-planner ros-humble-nav2-costmap-2d \
  ros-humble-nav2-lifecycle-manager ros-humble-nav2-msgs
Then re-source: source /opt/ros/humble/setup.bash
C++ app cannot connect to slam_bridge: "SLAM bridge not reachable on 127.0.0.1:9997" slam_bridge.py is not running, or it crashed before starting its TCP server. The TCP server starts in a daemon thread during __init__. Check slam_bridge.py is running: pgrep -af slam_bridge.
Check port 9997: ss -tlnp | grep 9997.
The SlamLink in the C++ app retries every 2s. If slam_bridge.py starts later (normal in start_tank.sh — it starts before the C++ app), the connection will establish automatically.
start_tank.sh startup order: 1. ydlidar driver (2s wait) → 2. slam_bridge.py (4s wait for SLAM Toolbox TF) → 3. EKF + navsat (if enabled) → 4. Nav2 planner (2s wait) → 5. mjpeg_bridge.py → 6. C++ app.
The delays ensure each subsystem is ready before the next starts.
